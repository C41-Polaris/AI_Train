第三章·训练流程全解析

从原始数据到可用的 LLM，中间经历了什么？本章拆解训练流水线的每一步。

3.1 数据准备

数据是模型能力的上限。质量 > 数量。

💡 核心原则

Garbage In, Garbage Out. 你投喂什么样的数据，模型就学会什么样的能力。数据清洗的时间应至少占项目总时间的 30-50%。

数据收集

  • Common Crawl：公开网页数据（~PB 级），LLaMA 和 GPT 都用过
  • GitHub Code：训练代码能力
  • ArXiv / 学术论文：训练推理和专业知识
  • Wikipedia：高质量百科知识
  • Books：长文本理解能力

数据清洗 Pipeline

Raw Data（原始数据，~50TB） ↓ 去重（MinHash + 精确匹配） → 去除 20-40% 的重复 ↓ 质量过滤（heuristic / 模型分类器） ↓ PII 脱敏（去除个人信息） ↓ Tokenization 统计 → 去除非自然语言（如过长 XML） ↓ 高质量文本（final）

📊 典型数据配比（LLaMA 3）

来源比例
网页~50%
代码~25%
书籍/论文~15%
数学/推理~10%

3.2 Tokenization——分词

计算机不认识"自然语言"，只认识数字。Tokenization 就是把文本切成一个个"词元"并映射为整数 ID。

BPE（Byte-Pair Encoding）

现代 LLM 默认的分词算法：

步骤: 1. 把文本分成 Unicode 字节 2. 统计相邻字节对的频率 3. 合并最频繁的对为一个新 token 4. 重复步骤 2-3 直到达到目标词表大小 例子: "低" "成本" "微" "调" "低成本" 出现 1000 次 → 合并为 [低成本] "微调" 出现 2000 次 → 合并为 [微调] 最终: 词表里有 [低成本] 和 [微调] 两个高频 token
  • ✅ 可以处理 OOV（未登录词）：任何词都能拆成已知子词/字节
  • ✅ 词表大小可控（通常 32K-128K）
  • ❌ 不同语言编码效率不同（中文约 1.5-2 tokens/字，英文约 0.3 tokens/字母）

特殊 Token

Token含义用途
<|endoftext|>文本结束分隔不同文档
<s> / </s>句子开始/结束序列边界
<pad>填充对齐序列长度
<unk>未知词未录入词表的词
<|im_start|>角色标记聊天模板分隔（如 ChatML）

3.3 Batching——批次组织

训练时不是一条一条送数据，而是以 batch 为单位：

Dataset: [文本₁, 文本₂, ..., 文本_N] 步骤 1: 随机打乱（shuffle） 步骤 2: 取 batch_size 条 步骤 3: 对每条做 tokenize 步骤 4: 在同一 batch 内做 padding（短的补到和最长的对齐） Batch 示例（batch_size=4, max_len=128）: ┌──────────────────────────────────────────────────┐ │ Token 1 │ Token 2 │ ... │ Token 128 │ Attn Mask │ ├──────────────────────────────────────────────────┤ │ 文本 A 完整 73 tokens │ 1 │ │ 文本 B 完整 128 tokens │ 1 │ │ 文本 C 被截断 ... │ 1 │ │ 文本 D 短文本 + <pad><pad>... │ 0 │ └──────────────────────────────────────────────────┘

关键参数：

  • batch_size：每次迭代喂多少条数据
  • gradient_accumulation_steps：累积 n 个 batch 再更新梯度（等效于增大 batch_size）
  • max_seq_length：序列最大长度（超过的被截断）

3.4 预训练（Pretraining）

预训练 = 让模型在大规模未标注文本上学习语言能力。核心任务只有一个：预测下一个 Token。

输入: "今天天气真" ↓ 模型预测下一个 token 的概率分布: [ "好": 0.65, "不": 0.12, "热": 0.08, "晴": 0.06, "棒": 0.05, ... ] ↓ 正确答案: "好" ↓ Loss = -log(0.65) = 0.43 ↓ 反向传播 → 更新参数

预训练的重要超参数

参数典型值说明
learning_rate3e-4 ~ 1e-4预训练常用较大的学习率
batch_size~4M tokensLLaMA 3 的 batch size
warmup_steps~2000学习率逐步增加到目标值的步数
weight_decay0.1权重衰减强度
num_epochs1-3预训练数据通常只过 1-2 遍
dtypebfloat16混合精度训练

预训练 vs 微调的对比

预训练微调
数据量TB 级（数万亿 tokens）MB-GB 级（数百~数万条）
计算量数千 GPU·天数小时~数天
学习率3e-4 ~ 1e-42e-5 ~ 5e-5（更低）

3.5 有监督微调（SFT）

预训练后的模型只会"续写文本"，不会回答问题。SFT 用 人工标注的问答对 教会模型遵循指令。

预训练模型: "Python 是一种" → "编程语言。" ← 只会续写 SFT 之后: "什么是 Python？" → "Python 是一种高级编程语言..." ← 会回答

SFT 数据格式

📝 ChatML 格式

<|im_start|>system
你是一个有用的 AI 助手。<|im_end|>
<|im_start|>user
什么是傅里叶变换？<|im_end|>
<|im_start|>assistant
傅里叶变换是一种数学变换，它可以将信号从时域转换到频域...<|im_end|>

3.6 人类偏好对齐: RLHF / DPO

SFT 让模型学会回答问题，但还没学会"什么回答更好"。RLHF 和 DPO 让模型与人类偏好对齐。

RLHF（基于人类反馈的强化学习）

步骤 1: 收集比较数据 提问: "如何减肥？" 回答 A: "控制饮食和运动。" 回答 B: "饿几天就行了。" 标注员选择: A > B ← 人类偏好 步骤 2: 训练 Reward Model（奖励模型） 用比较数据训练一个模型，学会判断"哪个回答更好" 步骤 3: 用 PPO 强化学习优化主模型 主模型生成回答 → Reward Model 打分 → PPO 更新主模型参数

DPO（Direct Preference Optimization）——更简单的方案

RLHF 需要训练一个额外的 Reward Model，实现复杂。DPO 直接利用偏好数据优化主模型。

DPO Loss = -log( σ( β · log(π(y_w|x)/π_ref(y_w|x)) - β · log(π(y_l|x)/π_ref(y_l|x)) ) )

其中 y_w 是偏好的回答，y_l 是不偏好的回答。简单理解：DPO 让模型更倾向于生成"被偏好的回答"的模式。

3.7 总结

📌 完整训练流水线

  1. 数据：收集 → 清洗 → 去重 → Tokenize
  2. 预训练：Next Token Prediction → 基座模型
  3. SFT：指令数据 → 对话模型
  4. 对齐（DPO/RLHF）：偏好数据 → 有用且安全的模型
