第六章·推理优化

训练是艺术，推理是工程。同样一个模型，经过优化可以快 10 倍、省 5 倍显存。

6.1 推理的瓶颈

💡 推理的两个阶段

自回归推理 = Prefill（预填充）+ Decode（逐词生成）。两个阶段有不同的瓶颈。

Prefill（一次） Decode（逐词，K 步） 输入: "用 Java 写一个二分" 第 1 步: 输出 "查" → 更新 KV Cache 第 2 步: 输出 "找" → 更新 KV Cache 并行计算所有输入 token 的 第 3 步: 输出 "算" → 更新 KV Cache attention 第 4 步: 输出 "法" → 更新 KV Cache 瓶颈: 计算密集（matrix multiply） 瓶颈: 访存密集（Memory Bound） 一次读完整个 prompt 每步只生成一个 token，重复从显存读取 KV Cache

一个 70B 模型推理的时间分布：

  • Prefill（假设 512 tokens）：~30% 总延迟
  • Decode（假设 256 tokens）：~70% 总延迟
  • 随着序列变长，Decode 占比越来越大

6.2 KV Cache

核心思想：自回归推理时，每生成一个新 token，都要计算它与之前所有 token 的 Attention。但之前 token 的 K 和 V 每次都是一样的——为什么不缓存起来？

第 1 步时（没有 Cache）: Q_new = Embed(tok₁) 计算 Attn(Q_new, K_all, V_all) → 重新算所有 K, V → 慢！ 第 2 步后（有 Cache）: Cache: K = [K₁, K₂], V = [V₁, V₂] ← 上次算过的 Q_new = Embed(tok₃) K_new = W_K · Q_new V_new = W_V · Q_new Attn(Q_new, [K_cache; K_new], [V_cache; V_new]) ↑ 用缓存的 K,V + 当前这一步新的 K,V 第 3 步后: Cache = [K₁, K₂, K₃], [V₁, V₂, V₃] ← 不断追加

KV Cache 的代价

KV Cache 显著加速推理，但代价是显存——序列越长，Cache 越大。

KV Cache 大小 = 2 × (num_layers × num_heads × d_head) × seq_len × precision_bits / 8

举例: LLaMA-3-7B (32层, 32 heads, d_head=128, FP16)
每 token: 2 × 32 × 32 × 128 × 2 = 524,288 bytes = 0.5 MB / token
4K 序列: 0.5 × 4096 = 2 GB
32K 序列: 0.5 × 32768 = 16 GB

KV Cache 优化方向：

  • MQA：多个 Query Head 共享同一组 KV，KV 量减为 1/h
  • GQA：折中方案，分组共享KV
  • KV Cache 量化：把 Cache 里的 K/V 量化到 INT8 甚至 INT4
  • PagedAttention：vLLM 的核心创新，类似虚拟内存管理 KV Cache

6.3 Speculative Decoding（推测解码）

标准自回归推理每步只能生成 1 个 token。Speculative Decoding 让你一次试探多个 token。

💡 类比

好比写代码时 IDE 的自动补全——先猜你接下来要打什么，猜对了直接接受，猜错了再自己打。

标准解码（每步生成 1 个 token）: 模型 → 生成 "你好" → 生成 "世界" → 生成 "！" Step 1 Step 2 Step 3 = 3 步 Speculative Decoding（用小模型做 draft）: 小模型快速 draft: "你好世界！"（轻量，快） 大模型校验: ✅ "你好" ✅ "世界" + ❌ "！"（实际应该是"。"） 大模型只做了 1 次前向传播，就校验了 3 个 token！ 其中 2 个正确 → 收获了 2 步的加速！
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-7B")
draft_model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-0.5B")

outputs = model.generate(
    inputs,
    assistant_model=draft_model,   # 用 0.5B 做 draft 模型
    max_new_tokens=256,
)

6.4 量化推理

推理侧的量化更常见：直接用 INT4/INT8 模型做推理。

GPU 推理显存需求: 模型 | FP16 | INT8 | INT4 ──────┼──────────────┼──────────────┼──────────── 7B | ~14 GB | ~7 GB | ~3.5 GB 13B | ~26 GB | ~13 GB | ~6.5 GB 34B | ~68 GB | ~34 GB | ~17 GB 70B | ~140 GB | ~70 GB | ~35 GB

6.5 主流推理框架

vLLM

vLLM 的核心创新是 PagedAttention——把 KV Cache 分页管理。

传统 KV Cache: 连续显存区域（固定大小），大量碎片浪费 请求1: [K₁|K₂|K₃|...|K₂₀₄₈|_|_|_|_] ← 预留完整长度 请求2: [K₁|K₂|_|_|_|...|_|_|_|_|_] ← 大量碎片浪费 PagedAttention: 分页缓存（按需分配 16-token/page） 请求1: Page[3]→Page[7]→Page[12]→Page[4] ← 逻辑连续，物理离散 请求2: Page[1]→Page[9] ✅ 没有碎片，显存利用率从 30% → 95% ✅ 支持 Copy-on-Write（多个请求共享前缀）

llama.cpp

纯 CPU 推理引擎，支持 GGUF 格式。

  • ✅ 纯 C++ 实现，无 GPU 依赖
  • ✅ 各种量化格式（Q2~Q8、IQ 系列）
  • ✅ 集成 OpenAI 兼容 API

TensorRT-LLM

NVIDIA 官方推理优化引擎，极高吞吐但需要 NVIDIA GPU。

6.6 总结

📌 推理优化检查清单

  1. 量化模型：先用 Q4_K_M 或 AWQ 4-bit，显存省 75%
  2. 用 vLLM：PagedAttention 大幅提升吞吐和显存利用率
  3. GQA 模型优先：选 LLaMA 3 / Qwen 2.5 等 GQA 模型，KV Cache 减半
  4. Speculative Decoding：有 draft 模型的话 2-3x 加速
  5. Flash Attention：长序列推理必备
  6. 连续批处理：服务多个请求时性能比逐条推理好得多
